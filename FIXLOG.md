- **Why the packed solvers were stuck:** roofline on the K=64 solve (60k systems, 23 ms): DRAM ~1 GB ≈ 2.8 ms, FLOPs 6 GFLOP ≈ 255 GFlops/s = 2% of peak → neither DRAM- nor FLOP-bound. The left-looking column loop issues **2·K block barriers + K serial dot chains** per system, the fwd/bwd solves add 2·K warp-serial reduction steps, and the packed-triangle scatter decode (isqrt + fixup loops, ~20 instr/element) is recomputed per element per system. That structure is why every micro-tweak (ILP, rank-1 right-looking, more warps, cusolverDn) measured ≈0 in 07-04c.
- **Fix: `cholesky_solve_tiled<K,NTH>`** (cholesky_kernels.cuh) — 16×16-tile blocked right-looking factorization, one system/block, stride-17 tile rows (bank-conflict-free column walks). Five stacked ideas, each measured on 60k synthetic SPD systems (bench_cholesky.cu, sm_86):
  1. *Blocked panels* (potrf16 | parallel TRSM rows | 2×2-register-blocked SYRK dots): barriers drop 2·K → ~3·K/16; trailing elements written once per PANEL not per column (kills the smem-store cost that sank rank-1 right-looking). → 1.16-1.37×
  2. *Forward solve folded into the panels* (bordered-matrix: b = one extra TRSM row + one SYRK dot per remaining row) — the serial fwd-solve phase disappears. → 1.31-1.50×
  3. *Inverse-diagonal backward solve*: each panel's TRSM also solves 16 identity rows = rows of L11⁻ᵀ (upper-triangular → stored in the never-read upper half of the diagonal tile, zero extra smem); bwd solve per tile-column = one parallel matvec instead of 16 warp-serial steps. (In-tile right-looking potrf also tried here: SLOWER, rank-1 loses even at tile scale.) → 1.34-1.54×
  4. *POTRF hidden under SYRK*: small SYRK-a updates tile(p+1,p+1) first, then warp 0 factors it WHILE the other warps run the bulk SYRK-b — the only remaining serial phase is hidden for every panel but the first. → 1.52-1.72×
  5. *Host-precomputed load map* (`build_cholesky_tile_map`, int2 {gmem ofs, smem ofs}, built once per K) + 2×2 SYRK register blocking (4 dots share 4 row-loads). → **1.86× / 1.93× / 2.08× / 2.26× at K=32/48/64/96** (6.6→3.6, 11.5→6.0, 22.8→10.9, 69.4→30.8 ms). ptxas: 56 regs, 0 spills.

## SYNC-2026-07-04d — packed solver at K=32 (headline: 5.25×→7.74×); solve-overlap pipeline; carveout lessons
- **The solve dominates at EVERY K, not just K≥48.** Honest per-batch timing at K=32 (Netflix, 3060): user Cholesky (`cholesky_solve_batched<32>`, thread-per-system) = **106.6 ms/iter = 72% of the APR iter**; true user WMMA is only ~14 ms (the old "120 ms User LHS+RHS / 750 GFlops/s" row was solve-polluted, same lie as the K≥48 rows fixed in 07-04c). External advice targeting the WMMA gather (L2 persistence, cp.async, __ldg etc.) chases a phantom — the gather is nearly free at all K.
- **Fix (big win): dispatch `cholesky_solve_packed<32>` instead of batched at K=32.** Batched streams a 4 KB/thread local matrix at ~42 GFlops/s; packed keeps the triangle in smem with one warp/system. User solve 106.6→**49.7 ms/iter** (2.14×). Netflix K=32 headline: **5.25×/4.69× → 7.74×/6.47×** (compute/wall), APR wall 5.14→3.40 s. ΔRMSE unchanged (test Δ 0.0266% vs 0.0265%; trajectories identical to 6 printed decimals). K=16 measured the OTHER way (batched 16.6 vs packed 21.5 ms/iter — tiny local matrix suits L1, packed wastes lanes) → new default `BATCHED_CHOLESKY_MAX_K=16` in common.cuh (was 32).
- **Solve/LHS dual-stream overlap pipeline** (double-buffered LHS, solve on high-priority stream sS, LHS+RHS on sC, phase-level ops on legacy NULL stream as implicit joins): implemented in both .cu, **bit-identical results** (baseline trajectories match old logs digit-for-digit at K=32/48/64/96), but measured **~no gain at K≥48**: solve (94.6 KB/SM smem at 11 blocks/SM) and WMMA (~8.4 KB/block, smem-bound) both saturate smem capacity, so blocks cannot co-reside, and two GPU-filling grids do NOT interleave on Ampere — "overlap" degenerates to time-slicing. K=64: 11.95×/10.37× → 11.95×/10.46× (noise). Keep the pipeline (free, and pays off automatically if the solve ever gets leaner, e.g. cuSolverDx), but do NOT expect stream overlap to beat smem-saturating kernel pairs.
- **Carveout lesson 1:** kernels with different L1/smem carveout configs cannot co-execute on Ampere. The scalar kernel's `cudaSharedmemCarveoutMaxL1` (07-04 spill mitigation) is vestigial since ROW_SPLIT (0-64 B stack) and forced two per-SM config drains per batch → replaced with uniform `MaxShared` on all pipeline kernels. - **Carveout lesson 2 (regression, caught & reverted):** forcing MaxShared on `cholesky_solve_batched` (zero smem, lives on L1 for its locals) cut its L1 to 28 KB and cost ~25% solve time at K=32 (137.8 vs 106.6 ms/iter). Batched solver now stays at driver default. Never force a smem carveout on local-memory-heavy kernels.
- **Dead scalar launches skipped:** in mixed mode, sparse jobs exist only for tile-rows ≥ tx_split, so batches fully inside the dense region launched a 1024-thread grid that 100% early-exited (7 of 9 Netflix user batches) — now skipped host-side.
- **WMMA dense kernel: register prefetch double-buffer (K≤48).** Tile t+1's col_indices/feature int4 gathers issue into registers before tile t's MMAs, hiding gather latency; single smem buffer, same bytes/order → bit-identical; sm_75-portable (no cp.async). ## SYNC-2026-07-04b — ROW_SPLIT: real fix for the K≥48 baseline spill cliff
- **FP32 baseline still non-physical at K≥48** (K32 5.3×, K48 **36.7×**, K64 **37.3×** compute "speedup"). The 07-03d RR_C fix only reduced the spill (44×→30×), it did not remove it. *Root cause, quantified:* one block covers a whole XB=32-row tile-row, so each thread persistently holds `RPT·RR_C·RC_C = K²/32` accumulator floats (K48: 96, K64: 192), but `__launch_bounds__(1024,1)` caps threads at 64 regs. ptxas -v on the RUNNING templates: K32 ⟨8,2,2⟩ 56 B stack/thread (fits L1) vs K48 ⟨8,3,3⟩ **280 B** and K64 ⟨32,2,2⟩ **656 B** → 280–670 KB/SM working set ≫ 96 KB L1 → every per-tile accumulator flush is an L2/DRAM round-trip. Traffic estimate K48: ~0.57 MB/tile-visit × 1.45 M tiles ≈ **830 GB/iter** ≈ 2.3 s at 3060 bandwidth; measured 3.16 s/iter. On top: `yb_eff` 256→128 at K>32 doubles the tile count (2× more flushes). Baseline collapsed 230→63 (K48) / 51 (K64) GFlops/s.
- **Fix: ROW_SPLIT** — split each tile-row across `gridDim.y = G` sibling blocks (K16:1, K32:2, K48:4, K64/96:8); each block keeps the full 1024-thread geometry but owns only 32/G rows, so RPT drops G-fold and the running templates become ⟨2..4, RR, RC⟩ with **0–64 B stack** (ptxas-verified). Per-row arithmetic order unchanged → numerically identical up to existing atomicAdd nondeterminism (verified: ml100k K48 trajectories match to ~1e-6). Cost: only the cooperative smem feature staging repeats per sibling block (L2-cached); segment reads are per-row (not duplicated), total atomic output volume unchanged. Kernel: `fused_kernels.cuh` (`DZ = blockDim.z`, `row_base = blockIdx.y·RPT·DZ`); hosts: `ROW_SPLIT` define + `rows_per_thread = XB/(DZ·ROW_SPLIT)` + `dim3(blocks, ROW_SPLIT)` in both `.cu`. Override per-build with `-DROW_SPLIT=n`.
- **Consequence for the thesis:** headline speedups at K≥48 must be re-measured against the fixed baseline; expect them to land near the K16/K32 range (~4–6×) instead of 30–37×.

## SYNC-2026-07-04
- **Scalar kernel missing `const __restrict__`** — WMMA kernels already had `const __restrict__` on read-only pointers, but the scalar kernel (`compute_LHS_RHS_BALS_block`) — the spill bottleneck — used bare `int*`/`float*`. Without `__restrict__`, compiler assumes `atomicAdd(&d_LHS_all[…])` may invalidate cached `seg_colidx`/`seg_values` reads → forces redundant reloads in the hot inner loop. Without `const`, read-only loads compete with spill data in L1 cache instead of routing through the dedicated read-only texture cache. **Fix:** added `const __restrict__` to all read-only pointers and `__restrict__` to output pointers in `fused_kernels.cuh`, `wmma_kernels.cuh`, `cholesky_kernels.cuh`. Effect: read-only data → texture cache (separate HW), L1 → exclusively for spill.
- **L1 cache not maximized for spill-heavy kernels** — on Ampere (sm_8x), L1 and shared memory share a 128 KB bank. Default partition wastes L1 on shared memory the kernel barely fills. **Fix:** added `cudaFuncAttributePreferredSharedMemoryCarveout = cudaSharedmemCarveoutMaxL1` for all scalar kernel templates in both `main_experiment.cu` and `justapr.cu`. GPU now picks the smallest shared-memory partition that satisfies the kernel's request, giving the rest (~64–96 KB) to L1 for spill data.

## SYNC-2026-07-03d (corrected)
- **FP32-baseline register-spill CLIFF at K≥48** — Netflix K-sweep showed a non-physical jump (K32 wall 4.7×, K48 wall **35.9×**, compute **43.8×**). *Problem:* old `RR_C=2` made the scalar-kernel thread-plane `TRp·TCp=(K/2)²` reach 576 (K48)/1024 (K64) → `DZ` collapses to 1 → `RPT=XB/DZ=32` → each thread holds `RPT·RR·RC = 192–384` accumulators. `__launch_bounds__(1024,1)` hard-caps physical regs at 64, so the rest spill to local memory (**ptxas -v: 2592 B spill stores** on the running RPT=32 template at K48).
